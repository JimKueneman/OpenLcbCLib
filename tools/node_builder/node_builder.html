Address Spaces

Address spaces define how other nodes read and write data on your node. Each space has a numeric ID and a range of addresses. Most are auto-configured from your Feature Flag selections — you only need to edit specific fields.
Configuration Description Information — read-only XML that describes the node's configurable options.
Configuration tools read this space to discover what settings a node supports and build their UI automatically. The content is a null-terminated UTF-8 XML document generated from the CDI file you load. The library stores the byte array in program memory and serves it to any node that requests it. The address range is set automatically when you load a CDI file.
This space is always present — it holds the CDI XML data.
CDI data is read-only; configuration tools read it to build their UI.
CDI space always starts at address zero.
Starting address (fixed at zero for CDI space).
Automatically set to CDI byte count minus 1 when a CDI file is loaded.
Optional text describing this address space to configuration tools.
A valid CDI XML file is required to set the address range. Click here to browse or drag & drop a CDI XML file.
All Memory — raw byte-level access to the entire node address range.
This space provides an unstructured, byte-addressable window into node memory. It is mainly used for debugging, diagnostics, or direct memory inspection by advanced configuration tools. Because the space is generic, the address range and read/write policy are user-configurable. Most nodes leave this enabled with read-only access so tools can peek at raw memory without risking corruption.
Whether this address space is advertised during memory configuration discovery.
If checked, this space can only be read, not written.
If checked, the space starts at the Low Address below instead of zero.
Starting address of this space. Enabled when Low Address Valid is checked.
Last byte address in this address space.
Optional text describing this address space to configuration tools.
Configuration Memory — writable space for user-configurable node settings.
This is where configuration tools read and write the settings described by the CDI. The memory layout starts with the ACDI user name and description at the User Name Address, followed by your application-specific configuration variables. The library maps CDI offsets to this space automatically. Set the Highest Address large enough to cover all your CDI-defined variables.
Configuration space is always present when Memory Configuration is enabled.
Configuration space is writable so tools can update node settings.
Automatically set when User Name Address is not zero.
Equals User Name Address when Low Address Valid is checked.
Last byte address in configuration space. Must be ≥ 126.
Optional text describing this address space to configuration tools.

ACDI Manufacturer — fixed virtual addresses for the CDI <identification> static fields.
Without this space a configuration tool would have to parse the CDI XML to locate the manufacturer name, model, and version strings. This space provides a shortcut: it maps the static <identification> data to well-known fixed addresses so tools can read them directly without any XML parsing. The values come from the SNIP panel and are stored as null-terminated strings at standard offsets defined by the OpenLCB specification.
ACDI Manufacturer space is always present — holds SNIP manufacturer data.
Manufacturer data is read-only; set via SNIP fields.
This space always starts at address zero.
Starting address (fixed at zero).
Fixed at 0x7C (124) — covers SNIP version + manufacturer name + model + HW/SW version.
Optional text describing this address space to configuration tools.
ACDI User — fixed virtual addresses for the user name and description fields.
Without this space a configuration tool would have to parse the CDI XML to find the configuration memory addresses for the user name and user description. This space provides a shortcut: it maps those fields to well-known fixed virtual addresses so tools can read and write them directly without any XML parsing. The library maps this space to the same backing store as the 0xFD configuration space at the User Name Address offset, so both spaces access the same physical data.
ACDI User space is present when Memory Configuration is enabled.
User data is writable — holds user name and description.
This space always starts at address zero.
Starting address (fixed at zero).
Fixed at 0x7F (127) — covers user version + user name + user description.
Optional text describing this address space to configuration tools.
Train FDI — read-only XML describing the train's function map for throttles.
The Function Description Information space works like CDI but for train nodes. Throttles read this XML to discover which functions a train supports (headlight, bell, horn, etc.) and how to display them. The content is a null-terminated UTF-8 XML document generated from the FDI file you load. The address range is set automatically when you load an FDI file.
Train FDI space is present when Train Control is enabled.
FDI data is read-only; configuration tools read it to discover train functions.
FDI space always starts at address zero.
Starting address (fixed at zero for FDI space).
Automatically set to FDI byte count minus 1 when an FDI file is loaded.
Optional text describing this address space to configuration tools.
A valid FDI XML file is required to set the address range. Click here to browse or drag & drop an FDI XML file.
Train Function Config — writable space for train function states (speed, lights, sound).
Throttles write to this space to control train functions such as speed, direction, headlight, bell, and horn. Each function occupies a fixed-size slot at a known offset. The FDI describes which functions exist and their offsets so throttles know where to write. Set the Highest Address large enough to cover all function slots your train node supports.
Train function config space is present when Train Control is enabled.
Function configuration is writable — throttles can modify function states.
This space always starts at address zero.
Starting address (fixed at zero).
Last byte address for train function configuration data.
Optional text describing this address space to configuration tools.
Firmware Upgrade — writable space for over-the-network firmware updates.
This space receives firmware image data during an upgrade. The upgrade protocol uses a freeze/write/unfreeze sequence: the upgrading tool freezes the space, streams the firmware binary in sequential write commands, then unfreezes to commit the image and reboot. The full 32-bit address range is advertised because the firmware tool manages addressing. Your Callbacks_freeze, Callbacks_write_firmware, and Callbacks_unfreeze implementations handle the platform-specific file I/O and flash operations.
Firmware space is present when Firmware Upgrade is enabled.
Firmware space is writable to allow over-the-network updates.
Firmware space always starts at address zero.
Starting address (fixed at zero).
Full 32-bit address range — firmware tools manage the actual range.
Optional text describing this address space to configuration tools.

Buffer Pool Sizing

The library pre-allocates fixed pools of message buffers at startup. Each value sets the maximum buffer depth for that pool. Larger pools handle more simultaneous traffic but use more RAM. The defaults work well for most nodes.
Maximum number of small buffers for protocol overhead and short CAN messages.
Maximum number of buffers for addressed point-to-point datagram messages.
Maximum number of buffers for Simple Node Information Protocol responses.
Maximum number of buffers for large streaming data transfers. 0 is fine if Streaming is not enabled.
Maximum number of virtual nodes your hardware can host. Train nodes each use one slot.

Callbacks

Callbacks are functions the library calls when specific events happen (message received, timer tick, login complete, etc.). Mandatory callbacks are always included. Optional callbacks appear based on which features you enabled.
