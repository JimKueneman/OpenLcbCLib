## [1.2.2] — 2026-03-16

### Added — Node Wizard / Node Config

- **Hover help for collapsible section headers** — the four `<details>` panels that
  were missing `data-help` attributes (Project Options, Well Known Events, Add-Ons,
  Advanced) now fire the right-panel help popup when hovered. Added HELP_DATA entries
  for all four keys with descriptions sourced from the relevant OpenLCB standards.

- **Help panel clears on mouse-leave** — added `_clearHelp()` and wired a `mouseleave`
  listener to every `[data-help]` element so the detail panel resets to the placeholder
  text when the cursor leaves a hotspot, instead of staying latched on the last item.

### Added — Node Wizard / Callbacks

- **Tile-level hover help** — hovering anywhere inside a callback group tile (but not
  on a specific checkbox row) now shows a group-level explanation in the right-hand
  detail panel. Moving the mouse off a checkbox row falls back to the group description
  rather than clearing; leaving the tile entirely clears to the placeholder.

- **`groupDetail` field on all 7 callback groups** — `callback_defs.js` now carries a
  `groupDetail` paragraph for each group (CAN, OLCB/LCC, Events, Config Memory,
  Firmware, Train, Broadcast Time) explaining what the group's checkboxes control,
  when each is typically needed, and references to the relevant OpenLCB standard.
  The existing short `description` strings used in the tile header are unchanged.

### Added — README

- **Standalone CDI / FDI Editor link** — the root `README.md` now includes a link to
  `cdi_fdi_wizard.html` (both the GitHub Pages URL and the local path) as a lighter
  tool for users who only need to author CDI or FDI XML without the full project
  generation workflow.

### Fixed — Node Wizard / CDI & FDI Editors

- **Stray text in container elements now flagged** — typing random characters (e.g.
  `llllll`) inside structural XML elements like `<segment>` or `<group>` was silently
  accepted because DOMParser treats it as valid text content. Both `cdi_validator.js`
  and `fdi_validator.js` now detect non-whitespace text nodes in container elements
  and report them as errors with correct line numbers.

- **Squiggle placement for line-less errors** — when a validation error has no line
  number, the CodeMirror linter now highlights the first line instead of a single
  invisible character at position 0.

  state, white copy button) instead of the previous dark theme.

  `tools/node_wizard/MAINTENANCE.md`. Covers all ten areas that need updating over

## [1.2.0] — 2026-03-16

### Added — Node Wizard / CDI Editor

- **`generate_cdi_schema.py`** — Python script that reads the OpenLCB CDI XSD
  and regenerates the `CDI_SCHEMA` block in `cm-adapter.js` automatically.
  Run whenever the CDI spec is updated to keep autocomplete in sync:
  ```
  python3 tools/node_wizard/vendor/codemirror/generate_cdi_schema.py \
      --xsd /path/to/cdi.xsd --patch
  ```
  The script replaces the block between `/* <<CDI_SCHEMA_START>> */` and
  `/* <<CDI_SCHEMA_END>> */` markers in `cm-adapter.js`.

- **CDI_SCHEMA updated to CDI 1.4** — the schema was previously hand-coded
  against an older version of the spec. Running `generate_cdi_schema.py`
  against the local CDI 1.4 XSD added the following previously missing items:
  `<action>` (action button), `<blob>` (raw data blob), `<link>` (URL reference),
  `<visibility>` and `<readOnly>` group hints, `<buttonText>` and `<dialogText>`
  action children, `<eventid>` map child, `<acdi>` fixed/var attributes,
  `<float>` size value `2`, `<float>` formatting attribute, and `<slider>`
  immediate attribute.

- **Schema-aware autocomplete** — CodeMirror 6 XML language is now configured with the full
  CDI element schema (`CDI_SCHEMA` in `cm-adapter.js`). Typing `<` or pressing Ctrl+Space
  inside any CDI element now offers only the elements and attributes that are valid at that
  position (e.g. `<segment>` only suggests `<name>`, `<description>`, `<group>`, `<int>`,
  `<float>`, `<string>`, `<eventid>`, `<bit>`). Attribute value completions are included where
  applicable (e.g. `size` on `<int>` suggests `1`, `2`, `4`, `8`).

- **Inline validation squiggles** — `@codemirror/lint` is now bundled and wired to
  `CdiValidator`. Errors appear as red underlines, warnings as dashed yellow, and info hints
  as dotted blue — 600 ms after the last keystroke. A lint-marker gutter column shows coloured
  dots opposite each affected line. The existing Validate button and footer panel are unchanged.

- **Format button** — a new `⇌ Format` toolbar button pretty-prints the XML in the editor
  with 2-space indentation. XML comments are preserved in place with their original internal
  formatting. The button is disabled when the editor is empty, matching the Validate button
  behaviour.

- **Duplicate singleton detection** in `CdiValidator` (Layer 2): reports an error if more
  than one `<identification>` or more than one `<acdi>` element appears as a direct child
  of `<cdi>`.

### Added — Node Wizard / FDI Editor

- **`generate_fdi_schema.py`** — Python script that reads the OpenLCB FDI XSD and regenerates
  the `FDI_SCHEMA` block in `cm-adapter.js` automatically. Run whenever the FDI spec is updated:
  ```
  python3 tools/node_wizard/vendor/codemirror/generate_fdi_schema.py \
      --xsd /path/to/fdi.xsd --patch
  ```
  The script replaces the block between `/* <<FDI_SCHEMA_START>> */` and
  `/* <<FDI_SCHEMA_END>> */` markers in `cm-adapter.js`.

- **Schema-aware autocomplete** — `createCMEditor()` now accepts an optional `schema` parameter;
  the FDI editor passes `FDI_SCHEMA` so autocomplete reflects FDI elements (`<fdi>`, `<segment>`,
  `<group>`, `<function>` with kind/size attributes, `<number>`, `<min>`, `<max>`, etc.).

- **Inline validation squiggles** — `FdiValidator` is now wired to the CodeMirror lint extension
  (identical to the CDI editor). Errors, warnings, and info hints appear as coloured underlines
  600 ms after the last keystroke.

- **Format button** — a new `⇌ Format` toolbar button (matching CDI) pretty-prints the FDI XML
  in the editor. Disabled when the editor is empty.

- **Toolbar updated to match CDI** — button naming and left-to-right order now matches the CDI
  editor: Open XML → Export XML → Generate Basic XML → separator → Validate → Format → Clear.

### Fixed — Node Wizard / FDI Editor

- **`FdiValidator` missing `analog` kind** — `VALID_KINDS` was `{'binary', 'momentary'}` which
  incorrectly flagged `kind="analog"` as an error. `analog` is now accepted.

- **`FdiValidator` unknown-tag warnings for `<min>` and `<max>`** — `<min>` and `<max>` are valid
  FDI children of `<function>` (for analog range) but were missing from `KNOWN_FDI_TAGS`, causing
  spurious *"Unknown element"* warnings. Both tags are now recognised.

- **`FdiValidator` missing singleton `<segment>` check** — the FDI XSD restricts the root `<fdi>`
  to exactly one `<segment>` child. A new Layer 2 check reports an error when more than one
  `<segment>` appears as a direct child of `<fdi>`.

### Fixed — Node Wizard / CDI Editor

- **Format wiped XML comments** — `formatXml()` in `cm-adapter.js` previously skipped all
  non-element, non-text DOM nodes, silently discarding every `<!-- ... -->` comment including
  the guidance blocks injected by *Generate Basic XML*. Comment nodes (nodeType 8) are now
  serialised with correct indentation and their internal text preserved verbatim.

## [1.1.0] — 2026-03-15

### Fixed
- **Consist forwarding silently dropped** — `ListenerAliasTable_initialize()` was not called
  at startup and `listener_find_by_node_id` was not wired into the TX state machine in
  `can_config.c`. Both DI pointer assignments are now guarded by `#ifdef OPENLCB_COMPILE_TRAIN`
  and added to `CanConfig_initialize()` and `_build_tx_statemachine()` respectively.
  Without this fix, all consist-forwarded Train Control messages were silently dropped because
  the alias-resolution callback was always NULL after the `memset()` initialisation.

### Changed
- macOS platform transport corrected from "Virtual CAN" to GridConnect in all documentation.

## [1.0.0] — 2026-03-15

Initial production release.

### Protocol support
- Full OpenLCB/LCC protocol stack in C
- Frame layer: alias negotiation, alias conflict detection, CAN framing
- Message layer: Verified Node ID, Protocol Identification (PIP), SNIP, Optional Interaction Rejected (OIR)
- Event protocol: producer/consumer, identify events, learn/teach
- Datagram protocol: send, receive, error handling
- Configuration memory protocol: read, write, lock, ACDI manufacturer and user spaces, CDI/FDI XML delivery
- Broadcast Time protocol: query, set, immediate time, rollover, producer and consumer roles
- Train Control protocol: speed, functions, emergency stop, global e-stop, global off, memory spaces, listener management, controller assignment, reserve/release, heartbeat

### Architecture
- No dynamic memory — all buffers statically defined at compile time
- No OS or RTOS dependency
- Dependency injection pattern for all hardware and platform callbacks
- Virtual node support with sibling loopback dispatch
- Periodic listener alias verification with configurable timing

### Platforms with working examples
- ESP32 + TWAI (Arduino IDE, PlatformIO)
- ESP32 + WiFi GridConnect (Arduino IDE, PlatformIO)
- Raspberry Pi Pico / RP2040 + MCP2517FD (Arduino IDE)
- STM32F4xx + CAN (STM32CubeIDE)
- TI MSPM0 + MCAN (Code Composer Theia)
- dsPIC + CAN (MPLAB X)
- macOS GridConnect (Xcode)

### Tools
- Node Wizard: browser-based project generator for all supported platforms
- OlcbChecker compliance test suite: frame, message, event, datagram, config memory, SNIP, CDI, FDI, broadcast time, train control (TR010–TR110)
