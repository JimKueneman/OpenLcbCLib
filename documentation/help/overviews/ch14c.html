Chapter 14c — Memory Configuration Protocol

The Memory Configuration Protocol provides remote read/write access to a node's configuration data through named address spaces. It is the primary mechanism for tools and throttles to configure nodes on the bus. The protocol is built on top of the Datagram Protocol (Ch 14) and optionally the Stream Transport Protocol (Ch 14b) for bulk transfers.

Source files: src/openlcb/protocol_config_mem_operations_handler.c/.h, src/openlcb/protocol_config_mem_read_handler.c/.h, src/openlcb/protocol_config_mem_write_handler.c/.h, src/openlcb/protocol_config_mem_stream_handler.c/.h

14c.1 Architecture: Four Handler Modules

The memory configuration implementation is split into four handler modules, each with its own callback interface:

flowchart TD D["Datagram Handler (Ch 14)"] --> OPS["Operations Handler
options, lock, freeze, reset"] D --> READ["Read Handler
per-space read dispatch"] D --> WRITE["Write Handler
write, write-under-mask"] D --> STREAM["Stream Handler
config-mem ↔ stream bridge"] STREAM --> ST["Stream Transport (Ch 14b)"] style D fill:#4a90d9,color:#fff style ST fill:#4a90d9,color:#fff

All four modules receive their commands as datagrams with the first byte set to 0x20 (memory configuration protocol identifier). The second byte identifies the specific command.

14c.2 Address Spaces

Each address space is identified by a single byte and represents a logically separate region of a node's configuration. See also Appendix C.

SpaceNameAccessDescription
0xFFCDIRead-onlyXML Configuration Description Information (Ch 16)
0xFEAllRead/WriteVirtual space combining all others
0xFDConfigRead/WriteUser-configurable data (primary config space)
0xFCACDI-MfgRead-onlyManufacturer info (SNIP fields 1-5)
0xFBACDI-UserRead/WriteUser name and description (SNIP fields 7-8)
0xFATrain FDIRead-onlyFunction Description Information XML (Ch 16b)
0xF9Train Fn ConfigRead/WriteTrain function configuration and state
0xEFFirmwareWrite-onlyFirmware upgrade data
Space encoding: Spaces 0xFD, 0xFE, and 0xFF have dedicated command byte variants for compactness. Other spaces use the generic "space in byte 6" encoding, where the space number appears as an extra byte in the datagram.

14c.3 Operations Handler

The Operations Handler manages node-level configuration operations that don't involve reading or writing data. All operations follow a command/reply pattern over datagrams.

14c.4 Read Handler

The Read Handler processes datagram-based memory read requests. Each read follows a two-phase pattern:

(data bytes) Tool-->>Node: Datagram Received OK

The handler dispatches to per-space read functions. For ACDI spaces (0xFC, 0xFB), it can optionally use the SNIP field loader functions to synthesize the response from the node's SNIP data structure.

14c.5 Write Handler

The Write Handler supports two write modes:

  • Plain write: Overwrites bytes starting at the given address.
  • Write-under-mask: A read-modify-write operation where alternating (mask, data) byte pairs are applied: new = (old & ~mask) | (data & mask).

Write-under-mask: The write handler requires a config_memory_read() callback to perform the read-modify-write cycle. If this callback is NULL, write-under-mask commands are rejected.

14c.6 Stream Handler (Config-Mem Stream Bridge)

The Stream Handler bridges the Memory Configuration Protocol with the Stream Transport Protocol (Ch 14b) for bulk transfers. This is used when transferring large address spaces (e.g., reading the entire CDI or uploading firmware) that would require many individual datagram round-trips.

The stream handler's ProtocolConfigMemStreamHandler_run() function must be called once per main loop iteration to pump data for active stream transfers. It checks for pending read data and sends it via the stream transport layer.

User streams: When the stream handler receives a stream initiate that is not a config-memory transfer, it forwards the request to the application's stream callbacks (user_on_initiate_request, etc.), allowing the application to use streams for custom purposes alongside config-mem streams.

14c.7 Command Byte Summary

All Memory Configuration commands share the datagram protocol identifier byte (0x20) as byte 0. The command byte (byte 1) determines the operation:

RangeCategory
0x00-0x0BWrite commands (plain and write-under-mask)
0x10-0x1BWrite replies (OK and fail)
0x20-0x23Write stream commands
0x30-0x3BWrite stream replies
0x40-0x43Read commands
0x50-0x5BRead replies (OK and fail)
0x60-0x63Read stream commands
0x70-0x7BRead stream replies
0x80-0xAAOperations (options, lock, freeze, reset, etc.)

14c.8 Callback Interfaces

Each of the four handler modules has its own callback interface. All interfaces require the datagram ACK/reject helpers. The per-space read/write callbacks are optional — NULL means the space is not supported.

Operations Handler

typedef struct {
    // REQUIRED: datagram helpers
    void (*load_datagram_received_ok_message)(...);
    void (*load_datagram_received_rejected_message)(...);

    // OPTIONAL: per-operation callbacks
    void (*operations_request_options_cmd)(...);
    void (*operations_request_get_address_space_info)(...);
    void (*operations_request_reserve_lock)(...);
    void (*operations_request_freeze)(...);
    void (*operations_request_unfreeze)(...);
    void (*operations_request_update_complete)(...);
    void (*operations_request_reset_reboot)(...);
    void (*operations_request_factory_reset)(...);
    void (*cleanup_before_handoff)(...);  // Called before freeze/reboot
} interface_protocol_config_mem_operations_handler_t;

Read Handler

typedef struct {
    // REQUIRED
    void (*load_datagram_received_ok_message)(...);
    void (*load_datagram_received_rejected_message)(...);
    uint16_t (*config_memory_read)(...);

    // OPTIONAL: per-space read handlers
    void (*read_request_config_definition_info)(...);  // 0xFF
    void (*read_request_all)(...);                     // 0xFE
    void (*read_request_config_mem)(...);              // 0xFD
    void (*read_request_acdi_manufacturer)(...);       // 0xFC
    void (*read_request_acdi_user)(...);               // 0xFB
    void (*read_request_train_function_config_definition_info)(...); // 0xFA
    void (*read_request_train_function_config_memory)(...);         // 0xF9
} interface_protocol_config_mem_read_handler_t;

Write Handler

typedef struct {
    // REQUIRED
    void (*load_datagram_received_ok_message)(...);
    void (*load_datagram_received_rejected_message)(...);
    uint16_t (*config_memory_write)(...);

    // OPTIONAL
    uint16_t (*config_memory_read)(...);  // Needed for write-under-mask
    // Per-space write handlers for 0xFF, 0xFE, 0xFD, 0xFC, 0xFB, 0xFA, 0xF9, 0xEF
    void (*on_function_changed)(...);     // Called when train function changed via 0xF9
} interface_protocol_config_mem_write_handler_t;

Stream Handler

typedef struct {
    // REQUIRED: transport
    void (*send_openlcb_msg)(...);
    void (*load_datagram_received_ok_message)(...);
    void (*load_datagram_received_rejected_message)(...);

    // Stream API
    stream_state_t* (*stream_initiate_outbound)(...);
    bool (*stream_send_data)(...);
    void (*stream_send_complete)(...);
    void (*stream_send_terminate)(...);

    // Per-space read/write callbacks
    // (one for each supported address space)

    // User stream callbacks (forwarded for non-config streams)
    bool (*user_on_initiate_request)(...);
    void (*user_on_initiate_reply)(...);
    void (*user_on_data_received)(...);
    void (*user_on_data_proceed)(...);
    void (*user_on_complete)(...);
} interface_protocol_config_mem_stream_handler_t;
