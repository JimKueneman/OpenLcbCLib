Chapter 17b — Train Search Protocol

The Train Search Protocol enables throttles to discover and allocate virtual train nodes by DCC address. A throttle broadcasts a search event containing the desired address, and any matching train node responds. If no match exists and the ALLOCATE flag is set, the command station creates a new virtual train node on demand.

Source files: src/openlcb/protocol_train_search_handler.c, src/openlcb/protocol_train_search_handler.h

17b.1 Search Event ID Encoding

Train search uses the well-known Event ID space 0x090099FF00000000. The lower 4 bytes encode the search query:

BitsFieldDescription
[63:32]Event space0x090099FF (fixed)
[31:8]Address digits6 nibbles (4 bits each): decimal digits 0-9, 0x0F = padding
[7:0]FlagsSearch options (see §17b.2)

The 6 nibbles encode the DCC address as a decimal digit string. For example, address 42 is encoded as 0F 0F 0F 0F 04 02 (four padding nibbles followed by digits 4 and 2). The utility function ProtocolTrainSearchHandler_digits_to_address() converts this back to a 16-bit integer.

17b.2 Flags

BitConstantMeaning
0x80TRAIN_SEARCH_FLAG_ALLOCATECreate a new virtual train node if no match found
0x40TRAIN_SEARCH_FLAG_EXACTRequire exact address match (no prefix matching)
0x20TRAIN_SEARCH_FLAG_ADDRESS_ONLYMatch by address only, not by name
0x08TRAIN_SEARCH_FLAG_DCCDCC protocol train
0x04TRAIN_SEARCH_FLAG_LONG_ADDRForce 14-bit long DCC address
0x03TRAIN_SEARCH_SPEED_STEP_MASKSpeed step mode (bits 1-0)

17b.3 Search Flow

The search is driven by the Event Transport Protocol. The throttle sends a Producer/Consumer Event Report carrying the search Event ID. The main dispatcher calls ProtocolTrainSearchHandler_handle_search_event() for each registered train node:

sequenceDiagram participant Throttle participant CS as Command Station participant T1 as Train Node (addr 42) participant T2 as Train Node (addr 99) Throttle->>CS: PCER: Search Event ID (address=42, EXACT|DCC) Note over CS: Dispatch to each train node CS->>T1: handle_search_event() Note over T1: Address matches! T1-->>Throttle: Producer Identified Set (search Event ID) CS->>T2: handle_search_event() Note over T2: Address 99 ≠ 42, skip

The matching train node responds with MTI_PRODUCER_IDENTIFIED_SET (0x0544), carrying the same search Event ID. The throttle now knows the Node ID of the train and can issue Train Control commands (Ch 17).

17b.4 No-Match / Allocate Flow

If no train node matches the search address and the ALLOCATE flag is set, the command station creates a new virtual train node:

sequenceDiagram participant Throttle participant CS as Command Station Throttle->>CS: PCER: Search Event ID (address=123, ALLOCATE|DCC) Note over CS: No existing train matches Note over CS: ALLOCATE flag set → create new node CS->>CS: on_search_no_match() callback Note over CS: Allocate virtual train node for addr 123 CS-->>Throttle: Producer Identified Set (search Event ID) Note over Throttle: New train node now available

The on_search_no_match() callback is called after the full train node enumeration completes with no match. The application allocates a new virtual train node and returns a pointer to it. The newly created node then sends the Producer Identified reply.

17b.6 Callback Interface

typedef struct {
    void (*on_search_matched)(        // A train node matches the search
        openlcb_node_t *node,         //   The matching train node
        uint16_t address,             //   Decoded DCC address
        uint8_t flags                 //   Search flags
    );
    openlcb_node_t* (*on_search_no_match)( // No match found, ALLOCATE set
        uint16_t address,                  //   Requested DCC address
        uint8_t flags                      //   Search flags
    );  // Returns pointer to newly allocated train node, or NULL
} interface_protocol_train_search_handler_t;
Compile guard: The Train Search handler requires both OPENLCB_COMPILE_TRAIN and OPENLCB_COMPILE_TRAIN_SEARCH to be enabled.
