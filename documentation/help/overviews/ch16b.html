Chapter 16b — Function Description Information (FDI)

Function Description Information (FDI) is the train-function counterpart to CDI (Ch 16). Where CDI describes a node's general configuration structure, FDI describes the functions available on a train node — headlight, bell, horn, smoke, etc. — as machine-readable XML. Configuration tools use FDI to present a meaningful UI for controlling and configuring train functions.

No dedicated handler: FDI is not implemented as a separate protocol handler. It is served as address space 0xFA through the Memory Configuration Protocol (Ch 14c). The XML content is provided by the application via config-memory read callbacks.

16b.1 What FDI Describes

An FDI document is an XML file that describes each function a train node supports. For each function, the FDI specifies:

  • Function number — the DCC function index (F0, F1, F2, ...)
  • Display name — human-readable label (e.g., "Headlight", "Bell")
  • Type — binary (on/off) or analog (variable)
  • Momentary flag — whether the function is momentary or latching

This allows throttles and configuration tools to show labeled buttons instead of generic "F0", "F1" identifiers.

16b.2 Address Space 0xFA

FDI occupies address space 0xFA (CONFIG_MEM_SPACE_TRAIN_FUNCTION_DEFINITION_INFO). Key properties:

16b.4 Reading FDI via Memory Configuration

FDI is accessed through the standard Memory Configuration read path. A configuration tool reads FDI the same way it reads CDI — by issuing read commands targeting space 0xFA:

sequenceDiagram participant Tool as Configuration Tool participant Train as Train Node Tool->>Train: Datagram: Read (space=0xFA, addr=0, count=64) Train-->>Tool: Datagram Received OK (reply pending) Train->>Tool: Datagram: Read Reply (first 64 bytes of FDI XML) Tool-->>Train: Datagram Received OK Note over Tool: Continue reading until null terminator...

For large FDI documents, the stream-based read path (command bytes 0x60) is more efficient, transferring the entire document in a single stream session. See Ch 14c §14c.6 for stream handler details.

16b.5 Train Function Config Space 0xF9

While FDI (0xFA) describes what functions exist, the Train Function Config space (0xF9) holds the actual runtime function configuration and state. This space is read/write and allows tools to:

  • Read current function states (on/off, analog values)
  • Write function configuration (custom labels, consist behavior settings)
PropertyValue
Space code0xF9
AccessRead/Write
PSI bitPSI_FUNCTION_CONFIGURATION (0x000040)
Write callbackon_function_changed() fires when a function value is modified

The relationship between these spaces mirrors CDI and Config: FDI is the schema (read-only XML), and Train Fn Config is the data (read/write bytes).
