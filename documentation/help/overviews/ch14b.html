Chapter 14b — Stream Transport Protocol

The Stream Transport Protocol provides reliable, flow-controlled transfer of arbitrarily large data between two nodes. Where datagrams are limited to 72 bytes, streams can transfer gigabytes using a sliding-window flow control mechanism with negotiated buffer sizes.

Source files: src/openlcb/protocol_stream_handler.c, src/openlcb/protocol_stream_handler.h

14b.1 Stream Lifecycle

Each stream progresses through three states. A node can have multiple concurrent streams open simultaneously (up to 255 per the specification).

stateDiagram-v2 [*] --> CLOSED CLOSED --> INITIATED : initiate_outbound() / receive Initiate Request INITIATED --> OPEN : receive Initiate Reply (accepted) INITIATED --> CLOSED : receive Initiate Reply (rejected) / TDE OPEN --> CLOSED : send/receive Complete or TDE note right of CLOSED : Slot available for reuse note right of INITIATED : Waiting for reply note right of OPEN : Data transfer active 
typedef enum {
    STREAM_STATE_CLOSED,     // Idle, slot not allocated
    STREAM_STATE_INITIATED,  // Waiting for Initiate Reply
    STREAM_STATE_OPEN        // Active data transfer
} stream_state_enum;

14b.2 MTI Reference

MTINameDirectionPurpose
0x0CC8Stream Initiate RequestSource → DestPropose a new stream with suggested buffer size and content UID

0x0888Stream Data ProceedDest → SourceFlow control: grant another window of bytes

14b.3 stream_state_t Structure

Each open or pending stream is tracked by a stream_state_t slot: 

typedef struct {
    stream_state_enum state;       // CLOSED, INITIATED, or OPEN
    uint8_t  source_stream_id;     // Source-assigned stream ID
    uint8_t  dest_stream_id;       // Destination-assigned stream ID
    node_id_t remote_node_id;      // Node ID of the remote end
    uint16_t remote_alias;         // CAN alias of remote (0 on non-CAN)
    uint16_t max_buffer_size;      // Negotiated window size in bytes
    uint32_t bytes_transferred;    // Total payload bytes sent so far
    uint16_t bytes_remaining;      // Bytes left in current send window
    bool     is_source;            // true = this node is the sender
    uint8_t  content_uid[6];       // Optional 6-byte stream content UID
    void    *context;              // Application context pointer
} stream_state_t;

14b.4 Initiate / Reply Negotiation

The source proposes a maximum buffer size. The destination may accept with a buffer size less than or equal to the proposed size, or reject the stream entirely.

sequenceDiagram participant Source participant Destination Source->>Destination: Stream Initiate Request (MTI 0x0CC8)
proposed buffer=4096, src_stream_id=1, content_uid Note over Destination: Evaluate request alt Accept Destination-->>Source: Stream Initiate Reply (MTI 0x0868)
buffer=2048 (≤ proposed), dest_stream_id=5, ACCEPT flag Note over Source,Destination: Stream OPEN — window = 2048 bytes else Reject Destination-->>Source: Stream Initiate Reply (MTI 0x0868)
error code, no ACCEPT flag Note over Source,Destination: Stream stays CLOSED end

The STREAM_REPLY_ACCEPT flag (0x8000) in the reply indicates acceptance. If this flag is absent, the reply carries a rejection error code.

14b.5 Flow Control: Data Send / Data Proceed

After the stream opens, the source sends data in chunks up to the negotiated buffer size. When the destination has consumed the data (or has buffer space available), it sends a Data Proceed message to grant another window:

sequenceDiagram participant Source participant Destination Note over Source,Destination: Stream OPEN, window = 2048 Source->>Destination: Stream Data Send (MTI 0x1F88) [chunk 1] Source->>Destination: Stream Data Send (MTI 0x1F88) [chunk 2] Note over Source: Window consumed Destination-->>Source: Stream Data Proceed (MTI 0x0888) Note over Source: Window reset to 2048 Source->>Destination: Stream Data Send (MTI 0x1F88) [chunk 3]

The source tracks bytes_remaining and stops sending when the window is exhausted. The Data Proceed message resets the window. For double-buffering, the destination can send an early proceed via ProtocolStreamHandler_send_early_proceed() before fully consuming the current window.

14b.6 Stream Completion and Error Handling

A stream ends in one of two ways:

  • Normal completion: Either side sends Stream Data Complete (MTI 0x08A8), optionally including the total byte count transferred.
  • Error termination: Either side sends Terminate Due to Error (MTI 0x00A8) with an error code. The stream slot is freed immediately.

The handler ProtocolStreamHandler_handle_terminate_due_to_error() matches the error to an open stream by checking the remote node and stream IDs, then frees the slot and fires the on_complete callback.

14b.7 Source-Side Outbound API

Applications that need to send data use the source-side API:

FunctionPurpose
ProtocolStreamHandler_initiate_outbound()Allocate a stream slot and send Initiate Request. Returns stream_state_t* or NULL if table full.
ProtocolStreamHandler_send_data()Send a chunk of data on an open stream. Returns false if window exhausted.
ProtocolStreamHandler_send_complete()Send Stream Data Complete to close the stream normally.
ProtocolStreamHandler_send_terminate()Send Terminate Due to Error with an error code.
ProtocolStreamHandler_send_early_proceed()Send an early Data Proceed for double-buffering.

14b.8 Callback Interface

The stream handler provides 5 optional callbacks for application-level stream handling: 

typedef struct {
    bool (*on_initiate_request)(     // Stream Initiate Request received
        openlcb_node_t *node,        //   (destination role)
        stream_state_t *stream       //   Return true to accept, false to reject
    );
    void (*on_initiate_reply)(       // Stream Initiate Reply received
        openlcb_node_t *node,        //   (source role)
        stream_state_t *stream
    );
    void (*on_data_received)(        // Stream data payload received
        openlcb_node_t *node,
        stream_state_t *stream,
        uint8_t *data, uint16_t len
    );
    void (*on_data_proceed)(         // Flow control ack received
        openlcb_node_t *node,
        stream_state_t *stream
    );
    void (*on_complete)(             // Stream completed or closed
        openlcb_node_t *node,
        stream_state_t *stream
    );
} interface_protocol_stream_handler_t;
Config Memory Streams: When the Memory Configuration Stream handler (Ch 14c) is enabled, it intercepts stream callbacks for config-memory transfers and forwards non-config streams to the application callbacks above.
