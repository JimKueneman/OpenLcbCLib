◆ CanBufferStore_initialize()

void CanBufferStore_initialize ( void )

Initializes the CAN Buffer Store.

Clears all message structures in the buffer pool, resets allocation flags, and zeros all telemetry counters. Must be called once during application startup before any buffer operations.

Use cases:

  • Called during system initialization
  • Required before any buffer allocation
Warning
MUST be called exactly once during initialization
NOT thread-safe
Attention
Call before CanBufferFifo_initialize()
Call before any CanBufferStore_allocate_buffer() operations
See also
CanBufferStore_allocate_buffer - Allocates from initialized pool

◆ CanBufferStore_allocate_buffer()

can_msg_t * CanBufferStore_allocate_buffer ( void )

Allocates a new CAN buffer from the pool.

Searches the buffer pool for an available buffer and returns a pointer to it. The buffer is cleared and marked as allocated. Allocation telemetry is updated to track current and peak usage.

Use cases:

  • Hardware CAN receive interrupt/callback
  • Creating outgoing CAN frames
  • Assembling multi-frame messages
Returns
Pointer to allocated CAN message buffer, or NULL if pool is exhausted
Warning
Returns NULL when pool exhausted - caller MUST check for NULL
NOT thread-safe
Attention
Always check return value for NULL before use
Caller must free buffer with CanBufferStore_free_buffer() when done
See also
CanBufferStore_free_buffer - Frees allocated buffer
CanBufferStore_messages_allocated - Current allocation count

◆ CanBufferStore_free_buffer()

void CanBufferStore_free_buffer ( can_msg_t * msg)

Frees a CAN buffer back to the pool.

Returns an allocated buffer to the pool for reuse. Marks the buffer as unallocated and decrements the allocation counter. NULL pointers are safely ignored.

Use cases:

  • After processing received CAN frame
  • After transmitting CAN frame
  • When discarding invalid message
Parameters
msgPointer to CAN message buffer to free (NULL is safely ignored)
Warning
NOT thread-safe - use shared resource locking
Attention
Do not use buffer after freeing
Safe to call with NULL pointer
See also
CanBufferStore_allocate_buffer - Allocates buffer
CanBufferFifo_pop - Returns buffer that must be freed

◆ CanBufferStore_messages_allocated()

uint16_t CanBufferStore_messages_allocated ( void )

Returns the number of CAN messages currently allocated.

Provides real-time count of allocated CAN message buffers. This is a live count that changes as buffers are allocated and freed.

Use cases:

  • Runtime monitoring of buffer usage
  • Detecting buffer leaks
  • Load balancing decisions
Returns
Number of CAN messages currently allocated (0 to USER_DEFINED_CAN_MSG_BUFFER_DEPTH)
Note
This is a live count that changes dynamically
Compare with messages_max_allocated for sizing analysis
See also
CanBufferStore_messages_max_allocated - Peak usage counter
CanBufferStore_allocate_buffer - Increments count
CanBufferStore_free_buffer - Decrements count

◆ CanBufferStore_messages_max_allocated()

uint16_t CanBufferStore_messages_max_allocated ( void )

Returns the maximum number of CAN messages that have been allocated.

Tracks peak allocation count since initialization or last reset. Useful for stress testing to determine optimal buffer pool size.

Use cases:

  • Sizing buffer pool during development
  • Stress testing and load analysis
  • Detecting memory usage spikes
Returns
Maximum number of CAN messages allocated simultaneously since last reset
Note
Peak value persists until CanBufferStore_clear_max_allocated() is called
Use during stress testing to optimize USER_DEFINED_CAN_MSG_BUFFER_DEPTH
See also
CanBufferStore_messages_allocated - Current allocation count
CanBufferStore_clear_max_allocated - Resets peak counter

◆ CanBufferStore_clear_max_allocated()

void CanBufferStore_clear_max_allocated ( void )

Resets the peak allocation counter to zero.

Clears the maximum allocation counter without affecting current allocations. Useful for starting fresh stress tests or monitoring sessions.

Use cases:

  • Starting a new stress test run
  • Monitoring specific test scenarios
  • Resetting telemetry after configuration changes
Warning
Does NOT affect currently allocated buffers
Attention
Current allocations remain unchanged
See also
CanBufferStore_messages_max_allocated - Returns peak value
