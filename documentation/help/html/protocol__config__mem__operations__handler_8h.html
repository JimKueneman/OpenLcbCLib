extern

Initializes the configuration memory operations protocol handler.

Sets up the protocol handler with the required callback interface for processing configuration memory operations commands. This must be called once during system initialization before any configuration operations can be processed.

The interface structure provides callbacks for datagram acknowledgment (required) and specific operation handlers (optional based on supported features).

Use cases:

  • Called once during application startup
  • Must be called before processing any configuration datagrams
Parameters
interface_protocol_config_mem_operations_handlerPointer to interface structure with callback functions
Warning
interface_protocol_config_mem_operations_handler must not be NULL
Required callbacks (load_datagram_received_ok_message, load_datagram_received_rejected_message) must be set
Attention
Call during initialization before enabling datagram reception
See also
interface_protocol_config_mem_operations_handler_t

Algorithm:

  1. Store pointer to interface structure in static variable
  2. Interface callbacks are now available for use by handler functions

The interface structure must remain valid for the lifetime of the application as the handler stores a pointer to it rather than copying its contents.

  • Called once during application startup
  • Must be called before processing any configuration datagrams

ProtocolConfigMemOperationsHandler_options_cmd

Generates an outgoing Get Configuration Options command datagram.

Algorithm:

  1. Load reply message header
  2. Copy CONFIG_MEM_OPTIONS_REPLY command byte to payload
  3. Build and copy available write flags to payload
  4. Build and copy available commands flags (high byte then low byte) to payload
  5. Copy write lengths byte to payload (hardcoded to 0x00)
  6. Set outgoing message as valid

This function generates a response to a Get Configuration Options query, advertising the node's configuration capabilities including supported commands, write operations, and address spaces.

Use cases:

  • Responding to configuration tool capability queries
  • Advertising node features during configuration discovery

ProtocolConfigMemOperationsHandler_get_address_space_info

Generates an outgoing Get Address Space Information command datagram.

Algorithm:

  1. Initialize description offset to 8 (default position)
  2. Load reply message header
  3. Check if requested space definition exists
  4. If space exists and is present:
    • Copy CONFIG_MEM_GET_ADDRESS_SPACE_INFO_REPLY_PRESENT command byte
    • Copy requested space identifier from incoming message
    • Copy highest address (4 bytes) to payload
    • Build and copy space info flags byte
    • If low address is valid:
      • Copy low address (4 bytes) to payload
      • Update description offset to 12
    • If description string exists (non-empty):
      • Copy description string to payload at description offset
    • Set outgoing message as valid
    • Return
  5. If space not present or doesn't exist:
    • Copy CONFIG_MEM_GET_ADDRESS_SPACE_INFO_REPLY_NOT_PRESENT command byte
    • Copy requested space identifier from incoming message
    • Set payload count to 8 (minimum size expected by OpenLcbChecker)
    • Set outgoing message as valid

This function responds to queries about specific address spaces, providing information about size, characteristics (read-only, low address), and description.

Use cases:

  • Responding to address space capability queries
  • Providing memory layout information to configuration tools

* 
* @param config_mem_operations_request_info Pointer to request information containing space definition

Generates an outgoing Lock/Reserve command datagram.

Creates and formats a Lock/Reserve command message to request exclusive access to another node's configuration. Used when acting as a configuration tool.

Use cases:

  • Requesting exclusive configuration access
  • Releasing configuration lock (with Node ID 0)

Generates an outgoing Lock/Reserve command datagram.

Algorithm:

  1. Load reply message header
  2. Extract requested Node ID from incoming message payload (6 bytes starting at position 2)
  3. Check current lock state:
    • If node is unlocked (owner_node == 0):
      • Set owner_node to requested Node ID (grant lock)
    • If node is already locked:
      • If requested Node ID is 0:
        • Release lock (set owner_node to 0)
      • Otherwise keep existing lock holder
  4. Load reply message header again (ensure clean state)
  5. Copy CONFIG_MEM_RESERVE_LOCK_REPLY command byte to payload
  6. Copy current owner_node (6 bytes) to payload at position 2
  7. Set outgoing message as valid

Lock behavior:

  • Unlocked node: Grant lock to any requester
  • Locked node: Only Node ID 0 can release lock
  • Lock holder is always returned in the reply

Use cases:

  • Granting exclusive configuration access
  • Releasing configuration lock
  • Checking current lock holder
